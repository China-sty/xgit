# git-ai 云端数据上传逻辑

## 概述

git-ai 客户端会将部分数据上报到服务端。默认上报目标为 `https://usegitai.com`，可通过 `~/.git-ai/config.json` 中的 `api_base_url` 配置项修改。

每个 HTTP 请求均携带 `X-Distinct-ID` 头（来自 `~/.git-ai/internal/distinct_id` 的唯一设备 UUID），用于标识安装实例。

---

## 配置项

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `api_base_url` | `https://usegitai.com` | API 服务端地址 |
| `api_key` | 无 | API 密钥，用于认证 |
| `prompt_storage` | `"default"` | Prompt 存储模式：`default`（上传 CAS）、`notes`（存 git notes）、`local`（不上传） |
| `default_prompt_storage` | 无 | 不受 `include_prompts_in_repositories` 管控的仓库的默认存储模式 |
| `notes_backend.kind` | `"git_notes"` | Notes 后端：`git_notes`（本地）或 `http`（上传远端） |
| `notes_backend.backend_url` | 无 | HTTP notes 后端的 URL |
| `telemetry_oss` | 无 | OSS 遥测开关，设为 `"off"` 关闭 Sentry/PostHog |
| `feature_flags.daemon_log_upload` | `true` | 是否上传守护进程日志 |
| `feature_flags.transcript_streaming` | `true` | 是否流式处理 AI 会话记录 |
| `feature_flags.transcript_sweep` | `true` | 是否定期扫描 AI 会话记录文件 |
| `disable_version_checks` | `false` | 关闭版本升级检查 |
| `disable_auto_updates` | `false` | 关闭自动更新 |

## 全部 API 端点一览

### 数据上报端点

| 端点 | 方法 | 数据内容 | 认证要求 | 触发条件 |
|------|------|----------|----------|----------|
| `/worker/cas/upload` | POST | AI 提示词全文 + 作者归属数据 | 自定义 URL 免认证 | `prompt_storage=default` 且有 CAS 数据待上传 |
| `/worker/cas/?hashes=...` | GET | 按 hash 读取 CAS 对象 | — | CAS 数据读取 |
| `/worker/metrics/upload` | POST | 使用统计、提交指标、会话事件 | 需要 `api_key` 或 OAuth | 有认证且有指标数据 |
| `/worker/logs/upload` | POST | 守护进程诊断日志 | 需要 `api_key` 或 OAuth | `daemon_log_upload=true` 且有认证 |
| `/worker/notes/upload` | POST | 作者归属笔记（commit_sha + content） | 需要 `api_key` 或 OAuth | `notes_backend.kind=http` 且有认证 |
| `/worker/notes/?commits=...` | GET | 按 commit SHA 读取 notes | — | `notes_backend.kind=http` |
| `/api/bundles` | POST | Prompt 记录 + 文件差异 | 需要 `api_key` 或 OAuth | 手动触发 |
| `/api/cube/*` | POST/GET | 分析查询（`git-ai analyze` 命令） | 需要 `api_key` | 运行 `git-ai analyze` |

### 认证端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/worker/oauth/device/code` | POST | OAuth 设备码流程 |
| `/worker/oauth/token` | POST | Token 交换/刷新 |

### 升级检查端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/worker/releases` | GET | 获取最新版本信息 |
| `/worker/releases/<channel>/download/SHA256SUMS` | GET | 下载校验和 |
| `/worker/releases/<channel>/download/install.sh` | GET | 下载安装脚本 |

## 认证门控逻辑

git-ai 对不同的 API 端点采用了**不同的认证策略**：

### 需要认证的端点（无认证则跳过上传）

`/worker/metrics/upload`、`/worker/logs/upload`、`/worker/notes/upload`、`/api/bundles`：

```rust
// 源码: src/api/metrics.rs
pub fn metrics_upload_allowed(_api_base_url: &str, client: &ApiClient) -> bool {
    client.is_logged_in() || client.has_api_key()
}
```

- 必须满足 `is_logged_in()`（OAuth 登录）**或** `has_api_key()`（配置了 API Key）中的一个
- 不满足条件时，telemetry_worker 直接跳过上传，不会发起 HTTP 请求
- **自定义 `api_base_url` 也不能绕过此检查**

### 自定义 URL 免认证的端点

`/worker/cas/upload`：

```rust
// 源码: src/daemon/telemetry_worker.rs
let using_default_api = api_base_url == crate::config::DEFAULT_API_BASE_URL;
if using_default_api && !client.is_logged_in() && !client.has_api_key() {
    // 仅在使用默认 API（usegitai.com）且未认证时跳过
    // 自定义 URL 不受此限制
    return;
}
```

- **如果配置了自定义 `api_base_url`，即使未认证也会上传 CAS 数据**
- 这是为了让自建服务端可以免认证接收数据

## 各端点数据格式

### 1. CAS Upload — `POST /worker/cas/upload`

**请求体** (`CasUploadRequest`)：

```json
{
  "objects": [
    {
      "hash": "<sha256>",
      "content": { /* AI prompt/response 的完整 JSON */ },
      "metadata": {}
    }
  ]
}
```

`CasObject.content` 包含 AI agent 的完整对话记录（`CasMessagesObject`），结构为 `{ "messages": [...] }`。

**响应体** (`CasUploadResponse`)：

```json
{
  "results": [
    { "hash": "<sha256>", "status": "ok" }
  ],
  "success_count": 1,
  "failure_count": 0
}
```

### 2. Metrics Upload — `POST /worker/metrics/upload`

**请求体** (`MetricsBatch`)：

```json
{
  "v": 1,
  "events": [
    {
      "t": 1234567890,
      "e": 1,
      "v": { "0": "value1", "1": 42 },
      "a": { "0": "attr1" }
    }
  ]
}
```

`MetricEvent` 使用稀疏数组编码（`SparseArray`），字段含义由 `event_id` 对应的 `EventValues` impl 决定。包含的数据类型：

- 提交统计（AI/人类代码行数、文件数）
- 检查点事件（checkpoint 调用次数、文件编辑量）
- 会话事件（AI agent 会话时长、模型信息）
- Agent 工具调用（MCP 调用、文件操作、命令执行）
- Rewrite 操作指标（stash/reset/rebase/cherry-pick）

**响应体** (`MetricsUploadResponse`)：

```json
{
  "errors": [
    { "index": 0, "error": "error message" }
  ]
}
```

### 3. Daemon Logs Upload — `POST /worker/logs/upload`

**请求体** (`DaemonLogsUploadRequest`)：

```json
{
  "version": 1,
  "git_ai_version": "1.6.16",
  "daemon_id": "<uuid>",
  "install_id": "<uuid>",
  "repo_url": "<optional>",
  "events": [
    {
      "id": "<uuid>",
      "kind": "diagnostic_event",
      "timestamp": "2026-07-21T06:50:00Z",
      "level": "info",
      "target": "git_ai::daemon",
      "message": "log message text",
      "fields": { "key": "value" },
      "repo_url": "<optional>",
      "git_ai_version": "1.6.16"
    }
  ]
}
```

日志级别：`trace`、`debug`、`info`、`warn`、`error`。

日志类型（`DaemonLogKind`）：
- `diagnostic_event`：普通诊断事件
- `heartbeat_event`：心跳事件（每 15 分钟发送一次，含 `uptime_seconds`、`os`、`arch`）

**响应体** (`DaemonLogsUploadResponse`)：

```json
{
  "accepted": 10,
  "dropped": 0,
  "enqueued": true,
  "errors": []
}
```

### 4. Notes Upload — `POST /worker/notes/upload`

**请求体** (`NotesUploadRequest`)：

```json
{
  "entries": [
    {
      "commit_sha": "<40-char hex>",
      "content": "<authorship-log-string>"
    }
  ]
}
```

`content` 字段为 `AuthorshipLog`（schema version `authorship/3.0.0`），包含：
- 文件级别的行归属（Agent session → 行号范围）
- Session 元数据（agent_id、model、human_author）
- Prompt 记录

**响应体** (`NotesUploadResponse`)：

```json
{
  "success_count": 1,
  "failure_count": 0
}
```

### 5. Notes Read — `GET /worker/notes/?commits=<sha1>,<sha2>,...`

**响应体** (`NotesReadResponse`)：

```json
{
  "notes": {
    "<sha>": "<authorship-log-string>",
    ...
  }
}
```

已知的 SHA 无 notes 时返回 `404` + `{"notes": {}}`。

### 6. CAS Read — `GET /worker/cas/?hashes=<sha256>,<sha256>,...`

**响应体** (`CAPromptStoreReadResponse`)：

```json
{
  "results": [
    {
      "hash": "<sha256>",
      "status": "ok",
      "content": { /* CasMessagesObject */ }
    }
  ],
  "success_count": 1,
  "failure_count": 0
}
```

### 7. Bundle — `POST /api/bundles`

**请求体** (`CreateBundleRequest`)：

```json
{
  "title": "bundle title",
  "data": {
    "prompts": { "<hash>": { /* PromptRecord */ } },
    "files": { "<path>": { /* ApiFileRecord */ } }
  }
}
```

## 第三方遥测（Sentry + PostHog）

除了向 `api_base_url` 上报外，git-ai 还集成了以下第三方遥测：

### Sentry

- **用途**：错误/崩溃报告、性能数据
- **DSN**：通过 `SENTRY_ENTERPRISE` 或 `SENTRY_OSS` 环境变量/编译时常量配置
- **关闭方式**：设置 `telemetry_oss: "off"`

### PostHog

- **用途**：使用行为分析
- **默认地址**：`https://us.i.posthog.com`
- **API Key**：通过 `POSTHOG_API_KEY` 环境变量配置
- **关闭方式**：设置 `telemetry_oss: "off"`

## 如何减少/关闭数据上传

在 `~/.git-ai/config.json` 中设置：

```json
{
  "prompt_storage": "local",
  "notes_backend": {
    "kind": "git_notes"
  },
  "telemetry_oss": "off",
  "feature_flags": {
    "daemon_log_upload": false,
    "transcript_streaming": false,
    "transcript_sweep": false
  },
  "disable_version_checks": true,
  "disable_auto_updates": true
}
```

- `prompt_storage: "local"` → 不上传 AI 提示词
- `notes_backend.kind: "git_notes"` → notes 只存本地 git notes
- `telemetry_oss: "off"` → 关闭 Sentry + PostHog
- `daemon_log_upload: false` → 不上传守护进程日志
- `transcript_streaming/sweep: false` → 不扫描/处理 AI 会话记录
- 不设置 `api_key`、不进行 OAuth 登录 → 大部分上传缺乏认证而被跳过
