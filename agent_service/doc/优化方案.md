# 🚀 Git AI Agent 企业级演进与优化方案

当前项目已经完成了从“硬匹配”到“向量语义检索”的核心跨越，并重构为了标准的 LangChain 模块化架构。
但为了支撑未来更多的指令、更复杂的问题以及满足企业级高并发部署的要求，我们需要在架构设计、基础设施和模型层进行系统性的演进。

以下是针对本项目演进为“企业级研发知识中枢”的详细优化方案。

---

## 1. 架构演进：从线性状态机到 Agentic Workflow (基于 LangGraph)

目前我们使用的是自研的 `SimpleAgent`（纯线性的 Extract -> Query -> Synthesize），缺乏动态决策能力。如果未来要支持“帮我查一下 XAppMode 的代码架构”或“这个 Bug 的历史修复 PR 是什么”，纯 RAG 链路将无法满足。

### 优化方向
- **引入 LangGraph / StateGraph**：将固定的三步流转升级为带有路由判断的有向无环图 (DAG)。
- **实现意图路由 (Intent Router)**：
  - 如果意图是 `find_owner`：走当前的 Milvus 埋点检索链路。
  - 如果意图是 `code_analysis`：调用 AST 工具或直接读取 Git 源码库。
  - 如果意图是 `chitchat`：直接走通用大模型回答。
- **引入反思与重试机制 (Reflexion)**：如果在 Milvus 中检索不到（返回了空），Agent 可以自动尝试扩大时间范围、同义词扩展后**再次发起检索**，而不是直接向用户返回“未找到”。

---

## 2. 基础设施升级：突破本地文件的性能瓶颈

目前的痛点：基于 SQLite 的并发读写冲突（经常容易 `database is locked`）以及 Milvus Lite 本地文件的内存与磁盘 I/O 限制。

### 优化方向
- **数据源解耦 (Kafka / RabbitMQ)**：
  - 将云端的日志收集由直接写 SQLite 改为写入消息队列（MQ）。
  - 同步脚本 `sync_vector.py` 变为常驻消费服务，实时监听 MQ，实现秒级增量同步。
- **分布式向量库 (Milvus Standalone / Cluster)**：
  - 弃用基于本地文件的 `milvus-lite`，部署真正的 Milvus 容器或集群服务。
  - 支持多路并发请求，解决目前 FastAPI 在高并发下由于争抢本地文件锁可能造成的性能剧降。
- **引入缓存层 (Redis)**：
  - 针对高频的热点提问（如“胶囊是谁负责”），在 `Query_Milvus_Node` 前增加 Redis 语义缓存层（如 GPTCache），命中直接返回，大幅节省大模型 Token 和响应时间。

---

## 3. 模型与提示词优化：进一步提升推理质量

虽然我们**已经实现了流式打字机响应 (Streaming)** 和**有向无环图 (DAG) 状态机路由**，并加入了 `Reflexion` 反思重试机制，但底层大模型总生成时间过长依然是瓶颈。

### 优化方向
- **模型大小分级调度**：
  - 对于简单的意图提取（如提取“胶囊”），使用速度极快、成本极低的小模型（如 `kimi-k1.5`、`qwen-turbo` 甚至本地小模型）。
  - 只有在最后的综合推理阶段，才动用 `kimi-k2.6` 这种重型模型进行深度阅读理解。
- **Prompt 工程极致精简**：
  - 目前大模型过于“啰嗦”，导致输出 Token 过多。
  - 优化 `synthesize_prompt`，使用 `Few-Shot` 限制其输出格式（例如严格强制输出 JSON 格式 `{"owner": "张三", "confidence": "high"}`），只在前端进行渲染，禁止其自行撰写寒暄语和冗长的分析过程。

---

## 4. RAG 检索质量优化：解决“找不准”的问题

虽然我们已经将 `values_json` 中的无关数字剔除了，但检索依然存在提升空间。

### 优化方向
- **混合检索 (Hybrid Search)**：
  - 纯向量检索在面对特定的工号、精确的文件名（如 `SystemUI.java`）时表现较差。
  - 在 Milvus 中引入 BM25 或结合 Elasticsearch 进行“向量语义 + 关键词精准”的双路召回 (Reciprocal Rank Fusion, RRF)。
- **Metadata 动态过滤**：
  - 在 `extract_entity_node` 中，不仅提取模块名，还要尝试提取时间限制（如“上周”）、人员限制。
  - 将这些限制作为 Milvus 的 `expr` 参数（标量过滤），例如 `expr="created_at > '2026-07-01'"`，大幅缩小检索范围并提高准确率。

---

## 5. 可观测性与企业级运维

- **完善 LangSmith 监控体系**：
  - 在 LangSmith 中设置自动化评估脚本（Evaluators），例如每天定时抽取 100 条线上对话，让 LLM-as-a-judge 自动评估准确率和召回率。
- **服务容器化部署 (Docker & K8s)**：
  - 编写 `Dockerfile` 和 `docker-compose.yml`。
  - 将 FastAPI 服务、Milvus 向量库、后台定时同步脚本全部容器化，配合 Kubernetes 实现高可用部署与故障自动重启。
