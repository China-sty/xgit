# 索引器介绍 (sync_vector.py)

`sync_vector.py` 是本 Agent 项目的“数据心脏”，它扮演着**数据清洗器**和**向量索引生成器**的双重角色。

当你执行 `python3 sync_vector.py` 命令时，系统会在后台经历一条严密的流水线，将冰冷的代码提交记录转化为大模型可以理解的“知识”。以下是完整的执行流程拆解：

---

## 1. 启动与防锁机制 (Anti-Lock)

* **动作**：脚本启动后，第一件事是寻找远端的 SQLite 原始数据库 `git_ai_local_analytics.db`。
* **防锁保护**：为了防止在读取时与正在写入的 Git 守护进程发生文件锁冲突（引发 `malformed` 或 `Permission denied` 错误），脚本会使用 `shutil` 将原始数据库**拷贝一份只读副本**（`temp_analytics_readonly.db`），所有的读取操作都在这个安全的副本上进行。

## 2. 增量扫描与时间戳控制 (Incremental Scan)

* **动作**：脚本会尝试读取同目录下的 `last_sync_time.txt` 文件。
* **分支逻辑**：
  * **第一次运行（冷启动）**：如果文件不存在，系统默认上次同步时间为 `1970-01-01`，此时会执行**全量同步**（或受限于 `LIMIT` 的冷启动提取）。
  * **后续运行（增量同步）**：如果文件存在，脚本提取其中的时间戳（例如 `2026-07-02 10:00:00`），并在 SQL 查询中加入 `WHERE created_at > ?`。**这意味着它只会捞出比上次同步时间更新的数据**，做到秒级扫描，0 Token 浪费。

## 3. 智能数据清洗与 Metadata 拆分 (Data Cleansing)

这是整个流水线中最核心的一环，决定了最终 Agent 回答的智商上限。

* **动作**：遍历查出的 SQLite 记录，对 JSON 格式的埋点数据进行“外科手术”式的拆解。
* **过滤垃圾埋点**：
  * 如果一条记录提取不出任何有意义的自然语言（如缺少 `files_changed` 或 `commit_message`），或者只包含纯数字，它将被**直接丢弃**。
* **分离 语义 (Page Content) 与 客观属性 (Metadata)**：
  * **Page Content (送去向量化的部分)**：将 `files_changed`（如 `CapsuleView.java`）和 `commit_message` 拼接成一句纯粹的业务描述文本。
  * **Metadata (不向量化，作为附加属性)**：将 `developer`（如 `shaoty1`）、`created_at`（提交时间）、`lines_added`（代码行数）打包成字典。
* **收益**：彻底断绝了类似 `"Unknown"` 这种无意义词汇污染向量空间导致大模型“瞎凑数”的 Bug。

## 4. 并发调用与向量化存储

清洗出高质量的自然语言文本后，系统需要调用阿里云大模型网关进行向量化：
1. **Batch 并发请求**：使用 `ThreadPoolExecutor` 开辟 10 个线程，将文本每 20 条切分为一个 Batch 进行并发请求。这使得原本串行需要数十秒的任务缩短至不足 1 秒。
2. **容错机制**：当 API 遇到超时、报错或网关限流时，会返回全零的占位向量 `[0.0] * 1024`，牺牲单条数据的精度，保全整个增量同步任务的大局不中断。
3. **Milvus Lite 存储**：将向量和对应的 `Metadata` 存入本地的 `milvus_demo.db`。这里强制配置了 `drop_old=False` 参数，以确保增量同步不会覆盖或删除之前耗费 Token 生成的历史知识。

## 5. 更新游标并退出 (Update Cursor)

* **动作**：脚本找出本次处理的数据中，最大的 `created_at` 时间戳。
* **落盘**：将这个最新时间戳覆写到 `last_sync_time.txt` 中。
* **结束**：在终端打印 `同步完成！增量数据已安全、干净地存入 Milvus Lite 中。`，进程正常退出。

---

**总结**：执行完这条命令后，你的 Agent 就像刚刚“吃透了”团队最新几天的代码提交记录一样，脑子里装满了最新的业务模块对应关系。随后，你就可以安心启动 `agent_demo.py` 去向它提问了！
